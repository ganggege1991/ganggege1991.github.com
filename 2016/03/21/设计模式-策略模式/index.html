策略模式(Strategy Pattern)

定义

策略模式定义了算法族，分别封装起来，让他们之间可以相互替换，此模式让算法独立于使用算法的客户。

“角色”

  • Context 上下文角色，起承上启下封装作用，屏蔽高层模块对策略，算法的直接访问，封装可能存在的变化。
  • Strategy 策略，算法的抽象。
  • ContcreteStrategy 具体实现。

类图（以简单的加减运算为例）

代码实现

/**
 * Strategy角色
 */
public interface Calculator{
    int exec(int a,int b);
}
/**
 * ContcreteStrategy角色
 */
public class Add implements Calculator{
    public int exec(int a,int b){
        return a+b;
    }
}
/**
 * ConcreteStrategy角色
 */
public calss Sub implements Calculator{
    public int exec(int a,int b){
        return a-b;
    }
}
/**
 * 上下文角色
 */
public class Context{
    private Calculator cal;
    public Context(Calculator cal){
        this.cal = cal;
    }
    public int exec(int a,int b){
        return cal.exec(int a,int b);
    }
}
//
public class Client{
    private static final String ADD_SYMBOL = "+";
    //private static final String SUB_SYMBOL = "-";
    public static void main(String[] args){
        Client.calculate(1,1,"+");
        Client.calculate(1,1,"-");
    }
    public static void calculate(int a ,int b,String symbol){
        Context con = null;
        if(ADD_SYMBOL.equals(symbol)){
            con = new Context(new Add());
        }else{
            con = new Context(new Sub());
        }
        return con.exec(a,b);
    }
}

优点

  • 算法自由切换
  • 避免使用多条件重复判断
  • 扩展性良好增加一个策略很容易，实现接口即可，其他不用修改

缺点

  • 策略类数量增多
  • 所有策略类都需要对外暴露

场景

  • 过个类只有算法或行为上稍有不同的场景
  • 算法需自由切换的场景
  • 需要屏蔽算法规则的场景

==注意事项==

如果策略实现类过多，可以考虑混合模式。

策略枚举

非常优秀和方便的模式，但是手枚举类型的限制，每一个枚举项都是，static,final,public的，扩展性收到了一定的限制，因此在系统开发中，枚举策略一般担当不经常发生变化的角色。

public enum Calculator {
    ADD("+") {
        @Override
        public int exec(int a, int b) {
            return a + b;
        }
    },
    SUB("-") {
        @Override
        public int exec(int a, int b) {
            return a - b;
        }
    };

    String symbol;

    private Calculator(String symbol) {
        this.symbol = symbol;
    }

    public abstract int exec(int a, int b);
}
